# Workspace Surface Audit

「このワークスペースとマシンが今実際に何ができるか、そして次に何を追加または有効化すべきか？」という問いに答えるための読み取り専用の監査スキルです。

これはセットアップ監査プラグインに対する ECC ネイティブの回答です。ユーザーが明示的にフォローアップの実装を依頼しない限り、ファイルを変更しません。

## 使用タイミング

- ユーザーが「set up Claude Code」「recommend automations」「what plugins or MCPs should I use?」または「what am I missing?」と言った場合
- より多くのスキル、フック、またはコネクタをインストールする前にマシンやリポジトリを監査する場合
- 公式マーケットプレイスプラグインと ECC ネイティブのカバレッジを比較する場合
- `.env`、`.mcp.json`、プラグイン設定、または接続アプリのサーフェスをレビューして欠落しているワークフロー層を見つける場合
- ケイパビリティがスキル、フック、エージェント、MCP、または外部コネクタのいずれであるべきか判断する場合

## 絶対ルール

- シークレット値を表示しないでください。プロバイダー名、ケイパビリティ名、ファイルパス、およびキーや設定が存在するかどうかのみを表示します。
- ECC が合理的にサーフェスを所有できる場合は、汎用的な「別のプラグインをインストール」アドバイスよりも ECC ネイティブのワークフローを推奨します。
- 外部プラグインはベンチマークとインスピレーションとして扱い、権威あるプロダクト境界としては扱いません。
- 以下の3つを明確に分離します:
  - 現在すでに利用可能
  - 利用可能だが ECC でうまくラップされていない
  - 利用不可で新しい統合が必要

## 監査入力

質問にうまく答えるために必要なファイルと設定のみを調査します:

1. リポジトリサーフェス
   - `package.json`、ロックファイル、言語マーカー、フレームワーク設定、`README.md`
   - `.mcp.json`、`.lsp.json`、`.claude/settings*.json`、`.codex/*`
   - `AGENTS.md`、`CLAUDE.md`、インストールマニフェスト、フック設定
2. 環境サーフェス
   - アクティブなリポジトリと明らかに隣接する ECC ワークスペースの `.env*` ファイル
   - `STRIPE_API_KEY`、`TWILIO_AUTH_TOKEN`、`FAL_KEY` などのキー名のみを表示
3. 接続ツールサーフェス
   - インストール済みプラグイン、有効化されたコネクタ、MCP サーバー、LSP、アプリ統合
4. ECC サーフェス
   - 既にニーズをカバーしている既存のスキル、コマンド、フック、エージェント、インストールモジュール

## 監査プロセス

### フェーズ 1: 存在するもののインベントリ

コンパクトなインベントリを作成します:

- アクティブなハーネスターゲット
- インストール済みプラグインと接続アプリ
- 設定済み MCP サーバー
- 設定済み LSP サーバー
- キー名から推定される env バックドサービス
- ワークスペースに関連する既存の ECC スキル

サーフェスがプリミティブとしてのみ存在する場合、それを指摘します。例:

- 「Stripe は接続アプリ経由で利用可能ですが、ECC には billing-operator スキルがありません」
- 「Google Drive は接続されていますが、ECC ネイティブの Google Workspace オペレーターワークフローがありません」

### フェーズ 2: 公式およびインストール済みサーフェスとのベンチマーク

ワークスペースを以下と比較します:

- セットアップ、レビュー、ドキュメント、デザイン、またはワークフロー品質に重複する公式 Claude プラグイン
- Claude や Codex にローカルインストールされたプラグイン
- ユーザーの現在接続されているアプリサーフェス

名前を列挙するだけでなく、各比較について以下に回答します:

1. 実際に何を行うか
2. ECC が既にパリティを持っているか
3. ECC がプリミティブのみ持っているか
4. ECC がそのワークフローを完全に欠落しているか

### フェーズ 3: ギャップを ECC の判断に変換する

すべての実際のギャップに対して、正しい ECC ネイティブの形を推奨します:

| ギャップタイプ | 推奨 ECC 形式 |
|-------------|--------------|
| 繰り返し可能なオペレーターワークフロー | スキル |
| 自動的な強制または副作用 | フック |
| 専門的な委譲された役割 | エージェント |
| 外部ツールブリッジ | MCP サーバーまたはコネクタ |
| インストール/ブートストラップガイダンス | セットアップまたは監査スキル |

ニーズがインフラ的ではなくオペレーション的な場合は、既存ツールをオーケストレーションするユーザー向けスキルをデフォルトとします。

## 出力フォーマット

以下の5セクションをこの順序で返します:

1. **現在のサーフェス**
   - 現在すでに利用可能なもの
2. **パリティ**
   - ECC が既にベンチマークに一致または超えている部分
3. **プリミティブのみのギャップ**
   - ツールは存在するが、ECC にクリーンなオペレータースキルがない
4. **欠落している統合**
   - まだ利用できないケイパビリティ
5. **トップ 3-5 の次のアクション**
   - 影響度順に並べた具体的な ECC ネイティブの追加

## 推奨ルール

- カテゴリごとに最も価値の高いアイデアを最大1-2つ推奨します。
- 明確なユーザー意図とビジネス価値を持つスキルを優先します:
  - セットアップ監査
  - 課金/顧客オペレーション
  - issue/プログラムオペレーション
  - Google Workspace オペレーション
  - デプロイ/オペレーション制御
- コネクタが企業固有の場合、ユーザーのワークフローに真に利用可能または明らかに有用な場合にのみ推奨します。
- ECC が既に強力なプリミティブを持っている場合、まったく新しいサブシステムを発明するのではなくラッパースキルを提案します。

## 良い成果

- ユーザーが何が接続されているか、何が欠落しているか、ECC が次に何を所有すべきかを即座に確認できます。
- 推奨事項が十分に具体的で、別の調査パスなしにリポジトリ内で実装できます。
- 最終的な回答は API ブランドではなくワークフロー中心に構成されています。
