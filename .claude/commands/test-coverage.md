# テストカバレッジ

テストカバレッジを分析し、ギャップを特定し、80%以上のカバレッジに達するために不足しているテストを生成します。

## ステップ 1: テストフレームワークの検出

| 指標 | カバレッジコマンド |
|------|-------------------|
| `jest.config.*` または `package.json` jest | `npx jest --coverage --coverageReporters=json-summary` |
| `vitest.config.*` | `npx vitest run --coverage` |
| `pytest.ini` / `pyproject.toml` pytest | `pytest --cov=src --cov-report=json` |
| `Cargo.toml` | `cargo llvm-cov --json` |
| `pom.xml` with JaCoCo | `mvn test jacoco:report` |
| `go.mod` | `go test -coverprofile=coverage.out ./...` |

## ステップ 2: カバレッジレポートの分析

1. カバレッジコマンドを実行
2. 出力を解析（JSONサマリーまたはターミナル出力）
3. **80%カバレッジを下回るファイル**をワースト順にリスト
4. カバレッジ不足の各ファイルについて、以下を特定:
   - テストされていない関数またはメソッド
   - 不足しているブランチカバレッジ（if/else、switch、エラーパス）
   - 分母を膨らませるデッドコード

## ステップ 3: 不足しているテストの生成

カバレッジ不足の各ファイルについて、以下の優先順位でテストを生成:

1. **ハッピーパス** — 有効な入力でのコア機能
2. **エラーハンドリング** — 無効な入力、データ不足、ネットワーク障害
3. **エッジケース** — 空の配列、null/undefined、境界値（0、-1、MAX_INT）
4. **ブランチカバレッジ** — 各if/else、switchケース、三項演算子

### テスト生成ルール

- テストはソースに隣接して配置: `foo.ts` → `foo.test.ts`（またはプロジェクト規約に従う）
- プロジェクトの既存のテストパターンを使用（インポートスタイル、アサーションライブラリ、モッキングアプローチ）
- 外部依存関係をモック（データベース、API、ファイルシステム）
- 各テストは独立 — テスト間で共有される可変状態なし
- テストを説明的に命名: `test_create_user_with_duplicate_email_returns_409`

## ステップ 4: 検証

1. 完全なテストスイートを実行 — すべてのテストが通ること
2. カバレッジを再実行 — 改善を確認
3. まだ80%を下回る場合、残りのギャップについてステップ3を繰り返し

## ステップ 5: レポート

前後の比較を表示:

```
カバレッジレポート
──────────────────────────────
ファイル                   前    後
src/services/auth.ts   45%     88%
src/utils/validation.ts 32%    82%
──────────────────────────────
全体:                   67%     84%  PASS:
```

## 焦点領域

- 複雑な分岐を持つ関数（高い循環的複雑度）
- エラーハンドラーとcatchブロック
- コードベース全体で使用されるユーティリティ関数
- APIエンドポイントハンドラー（リクエスト → レスポンスフロー）
- エッジケース: null、undefined、空文字列、空配列、ゼロ、負の数
