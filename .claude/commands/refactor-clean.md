# リファクタリング & クリーン

テスト検証付きでデッドコードを安全に特定・削除します。

## ステップ 1: デッドコードの検出

プロジェクトタイプに基づいて分析ツールを実行:

| ツール | 検出内容 | コマンド |
|--------|---------|---------|
| knip | 未使用のエクスポート、ファイル、依存関係 | `npx knip` |
| depcheck | 未使用のnpm依存関係 | `npx depcheck` |
| ts-prune | 未使用のTypeScriptエクスポート | `npx ts-prune` |
| vulture | 未使用のPythonコード | `vulture src/` |
| deadcode | 未使用のGoコード | `deadcode ./...` |
| cargo-udeps | 未使用のRust依存関係 | `cargo +nightly udeps` |

ツールが利用できない場合は、Grepを使用してインポートがゼロのエクスポートを検索:
```
# エクスポートを検索し、どこかでインポートされているか確認
```

## ステップ 2: 検出結果の分類

検出結果を安全性の階層で分類:

| 階層 | 例 | アクション |
|------|------|----------|
| **SAFE** | 未使用のユーティリティ、テストヘルパー、内部関数 | 自信を持って削除 |
| **CAUTION** | コンポーネント、APIルート、ミドルウェア | 動的インポートや外部コンシューマーがないことを確認 |
| **DANGER** | 設定ファイル、エントリーポイント、型定義 | 変更前に調査 |

## ステップ 3: 安全な削除ループ

SAFEな各アイテムについて:

1. **完全なテストスイートを実行** — ベースラインを確立（すべてグリーン）
2. **デッドコードを削除** — Editツールで外科的に除去
3. **テストスイートを再実行** — 何も壊れていないことを確認
4. **テストが失敗した場合** — `git checkout -- <file>` で即座に元に戻し、このアイテムをスキップ
5. **テストが通った場合** — 次のアイテムに進む

## ステップ 4: CAUTIONアイテムの処理

CAUTIONアイテムを削除する前に:
- 動的インポートを検索: `import()`、`require()`、`__import__`
- 文字列参照を検索: 設定内のルート名、コンポーネント名
- パブリックパッケージAPIからエクスポートされているか確認
- 外部コンシューマーがないことを確認（公開されている場合は依存先を確認）

## ステップ 5: 重複の統合

デッドコード削除後、以下を探す:
- ほぼ重複する関数（80%以上類似） — 1つに統合
- 冗長な型定義 — 統合
- 付加価値のないラッパー関数 — インライン化
- 目的を果たさないre-export — 間接参照を除去

## ステップ 6: サマリー

結果を報告:

```
デッドコードクリーンアップ
──────────────────────────────
削除:   12件の未使用関数
          3件の未使用ファイル
          5件の未使用依存関係
スキップ: 2件（テスト失敗）
節約:   約450行削除
──────────────────────────────
全テスト合格 PASS:
```

## ルール

- **テストを先に実行せずにコードを削除しない**
- **一度に1つの削除** — アトミックな変更でロールバックを容易に
- **不確実な場合はスキップ** — 本番を壊すよりデッドコードを残す方がよい
- **クリーニング中にリファクタリングしない** — 関心を分離（まずクリーン、次にリファクタリング）
