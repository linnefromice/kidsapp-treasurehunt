# ビルド & 修正

最小限で安全な変更により、ビルドエラーと型エラーを段階的に修正します。

## ステップ 1: ビルドシステムの検出

プロジェクトのビルドツールを特定してビルドを実行します:

| 指標 | ビルドコマンド |
|------|---------------|
| `package.json` に `build` スクリプト | `npm run build` または `pnpm build` |
| `tsconfig.json`（TypeScriptのみ） | `npx tsc --noEmit` |
| `Cargo.toml` | `cargo build 2>&1` |
| `pom.xml` | `mvn compile` |
| `build.gradle` | `./gradlew compileJava` |
| `go.mod` | `go build ./...` |
| `pyproject.toml` | `python -m compileall -q .` または `mypy .` |

## ステップ 2: エラーの解析とグループ化

1. ビルドコマンドを実行し、stderrをキャプチャ
2. ファイルパス別にエラーをグループ化
3. 依存関係の順序でソート（ロジックエラーの前にインポート/型を修正）
4. 進捗追跡のために合計エラー数をカウント

## ステップ 3: 修正ループ（一度に1つのエラー）

各エラーについて:

1. **ファイルを読む** — Readツールを使用してエラーのコンテキスト（エラー周辺10行）を確認
2. **診断** — 根本原因を特定（インポート不足、型の不一致、構文エラー）
3. **最小限に修正** — Editツールでエラーを解決する最小の変更を適用
4. **ビルドを再実行** — エラーが消え、新しいエラーが導入されていないことを確認
5. **次に進む** — 残りのエラーに対して続行

## ステップ 4: ガードレール

以下の場合はユーザーに確認:
- 修正が**解決するよりも多くのエラーを導入**する場合
- **3回の試行後も同じエラーが継続**する場合（より深い問題の可能性）
- 修正に**アーキテクチャの変更**が必要な場合（単なるビルド修正ではない）
- ビルドエラーが**依存関係の不足**に起因する場合（`npm install`、`cargo add` などが必要）

## ステップ 5: サマリー

結果を表示:
- 修正したエラー（ファイルパス付き）
- 残りのエラー（ある場合）
- 導入された新しいエラー（ゼロであるべき）
- 未解決の問題に対する推奨される次のステップ

## リカバリ戦略

| 状況 | アクション |
|------|----------|
| モジュール/インポートの不足 | パッケージがインストール済みか確認し、インストールコマンドを提案 |
| 型の不一致 | 両方の型定義を読み、より狭い型を修正 |
| 循環依存 | インポートグラフでサイクルを特定し、抽出を提案 |
| バージョン競合 | `package.json` / `Cargo.toml` のバージョン制約を確認 |
| ビルドツールの設定ミス | 設定ファイルを読み、動作するデフォルト値と比較 |

安全のため一度に1つのエラーを修正。リファクタリングよりも最小限の差分を優先してください。
