# Asideコマンド

タスク中に質問をすると、即座に焦点を絞った回答が得られ、そのまま中断した場所から続行します。現在のタスク、ファイル、コンテキストは一切変更されません。

## 使用タイミング

- Claudeが作業中に何か気になることがあり、勢いを失いたくない場合
- Claudeが現在編集しているコードの素早い説明が必要な場合
- タスクを脱線させずに決定に対するセカンドオピニオンや確認が欲しい場合
- Claudeが進む前にエラー、概念、パターンを理解する必要がある場合
- 新しいセッションを開始せずに現在のタスクとは無関係な質問をしたい場合

## 使用方法

```
/aside <あなたの質問>
/aside この関数は実際に何を返しますか？
/aside このパターンはスレッドセーフですか？
/aside ここでYの代わりにXを使っているのはなぜですか？
/aside foo()とbar()の違いは何ですか？
/aside 追加したN+1クエリを心配すべきですか？
```

## プロセス

### ステップ 1: 現在のタスク状態をフリーズ

何かに回答する前に、以下を心に留めます:
- アクティブなタスクは何か？（どのファイル、機能、問題に取り組んでいたか）
- `/aside` が呼び出された時点でどのステップが進行中だったか？
- 次に何が起こる予定だったか？

aside中にファイルを触る、編集する、作成する、削除することは絶対にしないでください。

### ステップ 2: 質問に直接回答

完全で有用でありながら最も簡潔な形で質問に回答します。

- 推論ではなく回答を先に
- 短く保つ — 完全な説明が必要な場合はタスク後に詳しく説明することを提案
- 質問が現在のファイルまたは作業中のコードについてであれば、正確に参照（関連する場合はファイルパスと行番号）
- 回答にファイルの読み取りが必要な場合は読む — ただし読み取りのみ、決して書き込まない

レスポンスのフォーマット:

```
ASIDE: [質問を簡潔に再述]

[回答]

— タスクに戻る: [何をしていたかの1行説明]
```

### ステップ 3: メインタスクを再開

回答の後、一時停止した正確なポイントからアクティブなタスクを即座に続行します。asideの回答がブロッカーや現在のアプローチを再考する理由を明らかにしない限り（エッジケースを参照）、再開の許可を求めないでください。

## エッジケース

**質問が提供されていない（`/aside` の後に何もない）:**
回答:
```
ASIDE: 質問が提供されていません

何を知りたいですか？（質問すると現在のタスクコンテキストを失わずに回答します）

— タスクに戻る: [何をしていたかの1行説明]
```

**質問が現在のタスクの潜在的な問題を明らかにした場合:**
再開前に明確にフラグを立てる:
```
ASIDE: [回答]

WARNING: 注意: この回答は現在のアプローチに[問題]があることを示唆しています。続行する前にこれに対処しますか、それとも計画通り進めますか？
```
再開前にユーザーの判断を待ちます。

**質問が実際にはタスクのリダイレクトである場合（サイド質問ではない）:**
質問が構築内容の変更を示唆している場合（例: `/aside 実は、代わりにRedisを使いましょう`）、確認:
```
ASIDE: これはサイド質問ではなく、方向転換のようです。
以下のどちらを希望しますか：
  (a) 情報としてのみ回答し、現在の計画を維持
  (b) 現在のタスクを一時停止してアプローチを変更
```
ユーザーの回答を待ちます — 仮定しないでください。

**質問が現在開いているファイルやコードについての場合:**
ライブコンテキストから回答します。セッション内で以前にファイルを読んだ場合は直接参照。そうでなければ今読み（読み取り専用）、ファイル:行の参照で回答します。

**アクティブなタスクがない（`/aside` 呼び出し時に何も進行中でない）場合:**
レスポンスの形を一貫させるために標準ラッパーを使用:
```
ASIDE: [質問を簡潔に再述]

[回答]

— タスクに戻る: 再開するアクティブなタスクなし
```

**質問が長い回答を必要とする場合:**
本質的な回答を簡潔に与え、以下を提案:
```
短いバージョンです。[現在のタスク]を終えた後に詳しい説明が必要ですか？
```

**連続で複数の `/aside` 質問がある場合:**
各質問に順番に回答します。最後の回答後にメインタスクを再開します。一連のasideを通じてタスク状態を失わないでください。

**asideの回答がコード変更が必要であることを示唆する場合:**
必要な変更に注意しますが、aside中には行わない:
```
ASIDE: [回答]

修正の価値あり: [何を変更すべきか]。現在のタスクの後にこれをフラグ付けします。今すぐ対処したい場合を除きます。
```

**質問が曖昧すぎるまたは漠然としている場合:**
1つの明確化質問を尋ねます — 回答に必要な情報を得るための最短の質問。複数の質問は尋ねないでください。

## 出力例

```
User: /aside fetchWithRetry()は実際に何をしますか？

ASIDE: fetchWithRetry()は何をしますか？

fetchWithRetry()（src/api/retry.ts:12）は、指数バックオフ（250ms → 500ms → 1s）で
最大3回リクエストを試みます。5xxレスポンスとネットワークエラーのみリトライし、
4xxエラーは最終的なものとしてリトライしません。

— タスクに戻る: src/middleware/auth.tsの認証ミドルウェアのリファクタリング
```

---

## 注意事項

- aside中にファイルを変更しない — 読み取り専用アクセスのみ
- asideは会話の一時停止であり、新しいタスクではない — 元のタスクは常に再開される必要がある
- 回答は焦点を絞る: 目的はユーザーを素早くアンブロックすること、講義を提供することではない
- asideがより大きな議論を生んだ場合、asideがブロッカーを明らかにしない限り、まず現在のタスクを完了する
- asideはタスクの結果に明確に関連しない限り、セッションファイルに保存されない
