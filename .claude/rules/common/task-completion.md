# タスク完了チェックリスト

コードを変更する実装タスクの完了前に適用。

---

## 完了条件

コードを変更した場合、以下の全ステップを完了するまでタスクは未完了とする。

### Step 1: 品質チェック

プロジェクトの品質チェックコマンドを順番に実行する。

<!-- プロジェクトに応じてコマンドを調整 -->
<!-- 例: lint, format, typecheck, build, test -->

| 順序 | コマンド | 目的 |
| ---- | -------- | ---- |
| 1 | lint（自動修正あり） | Lint チェック・修正 |
| 2 | format | フォーマット |
| 3 | typecheck | 型チェック |
| 4 | build | ビルド検証 |
| 5 | test | テスト全通過 |

失敗時はエラーを修正して再実行する。

### Step 2: ドキュメント更新確認

変更が機能・画面・API に影響する場合、関連ドキュメントの更新余地を確認する。

<!-- プロジェクトに応じてドキュメントパスを記載 -->

以下に該当する変更がある場合、ドキュメントを更新すること:

- 画面の追加・削除・URL変更
- API の追加・変更
- 入力項目・バリデーションルールの変更
- ステータス遷移・アクションの追加・変更

### Step 3: コミット

- `git status` で変更内容を確認
- `git add` で変更ファイルをステージング（`.ai/` は絶対に含めない）
- `git-workflow.md` の Conventional Commits 形式でコミット

### Step 4: プッシュ & PR

デフォルトブランチ以外で作業している場合:

- `git push -u origin <branch>` でリモートに反映
- `gh pr create` でPR作成（未作成の場合）
- PR URLをユーザーに報告

---

## 適用外

以下の場合はこのチェックリストをスキップできる:

- 調査・分析のみのタスク（コード変更なし）
- ユーザーが明示的に「コミットしないで」「チェック不要」と指示した場合
- `/pipeline` コマンド経由の実行（pipeline 自身が同等のフローを持つ）
