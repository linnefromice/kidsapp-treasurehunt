# 開発ワークフロー

> このファイルは [common/git-workflow.md](./git-workflow.md) を、Git操作の前に行う完全な機能開発プロセスで拡張します。

機能実装ワークフローは、開発パイプライン（調査、計画、TDD、コードレビュー、そしてGitへのコミット）を記述します。

## 機能実装ワークフロー

0. **調査と再利用** _（新しい実装の前に必須）_
   - **まず GitHub コード検索:** 新しいものを書く前に、`gh search repos` と `gh search code` を実行して既存の実装、テンプレート、パターンを見つけます。
   - **次にライブラリドキュメント:** Context7 またはベンダーの公式ドキュメントを使用して、実装前に API の動作、パッケージの使用方法、バージョン固有の詳細を確認します。
   - **最初の2つが不十分な場合のみ Exa を使用:** GitHub 検索と公式ドキュメントの後に、より広範なWeb調査や発見のために Exa を使用します。
   - **パッケージレジストリを確認:** ユーティリティコードを書く前に、npm、PyPI、crates.io などのレジストリを検索します。手書きのソリューションよりも実績のあるライブラリを優先します。
   - **適用可能な実装を検索:** 問題の80%以上を解決し、フォーク、移植、またはラップできるオープンソースプロジェクトを探します。
   - 要件を満たす場合は、ゼロから新しいコードを書くよりも、実証済みのアプローチの採用または移植を優先します。

1. **まず計画**
   - **planner** エージェントで実装計画を作成する
   - コーディング前に計画ドキュメントを生成: PRD、アーキテクチャ、system_design、tech_doc、task_list
   - 依存関係とリスクを特定する
   - フェーズに分解する

2. **TDD アプローチ**
   - **tdd-guide** エージェントを使用する
   - まずテストを書く（RED）
   - テストをパスする実装を行う（GREEN）
   - リファクタリング（IMPROVE）
   - 80%以上のカバレッジを確認する

3. **コードレビュー**
   - コードを書いた直後に **code-reviewer** エージェントを使用する
   - CRITICAL と HIGH の問題に対処する
   - 可能な限り MEDIUM の問題を修正する

4. **コミット＆プッシュ**
   - 詳細なコミットメッセージを書く
   - Conventional Commits 形式に従う
   - コミットメッセージ形式と PR プロセスについては [git-workflow.md](./git-workflow.md) を参照

5. **レビュー前チェック**
   - すべての自動チェック（CI/CD）がパスしていることを確認する
   - マージコンフリクトを解決する
   - ブランチがターゲットブランチの最新状態であることを確認する
   - これらのチェックがパスした後にのみレビューを依頼する
